# Localization

ScavLib's localization layer translates your item names, keybind labels, and any
other UI strings into the player's language. You've already used it without noticing:
the dictionary overloads on `CustomItemBuilder.DisplayName` / `Description` and
`KeyBindBuilder.DisplayName` / `Description` feed straight into it. This page covers
that path directly, plus the JSON-file route for translators, and the helper that
generates a translation template.

There are two ways to supply translations: **inline dictionaries** in code, and
**JSON language files** on disk. They coexist — use inline strings for a handful of
labels, JSON files when you want translators to work outside your source.

## Inline strings

A translation is just a dictionary from a **language code** to text. The builders
take one directly:

```csharp
var names = new Dictionary<string, string>
{
    { "EN", "Old Flask" },
    { "ZH", "旧酒壶" },
};

CustomItemBuilder.Canteen("mymod_flask", "MyMod")
    .DisplayName(names)
    .Register();
```

For strings that aren't an item or keybind label, register them directly under a key
and read them back through the game's locale lookup:

```csharp
using ScavLib.i18n;

LocaleManager.RegisterString("mymod_greeting", new Dictionary<string, string>
{
    { "EN", "Hello!" },
    { "ZH", "你好!" },
});

// elsewhere:
string text = Locale.GetOther("mymod_greeting");
```

Under the hood, item display names go through `LocaleManager.RegisterItem` and
keybind labels through `RegisterString` — you rarely call those yourself, but they're
public if you need them.

!!! tip "Always provide an `EN` entry"

    When the player's language isn't in your dictionary, ScavLib falls back in order:
    the current language code → `EN` → `English` → whatever the first entry is. An
    `EN` key guarantees a sensible fallback for every language you didn't translate.

## JSON language files

For larger mods, ship translations as JSON so translators never touch code. Put one
file per language in a `lang/` folder inside your mod, then register the folder:

```text
MyMod/
  lang/
    EN.json
    ZH.json
```

```csharp
LocaleManager.AutoRegister("MyMod", modFolderPath);
```

`AutoRegister` always loads `EN.json`, then the file matching the current game
language (a `zh-CN` game locale also tries `zh`). Each file is a JSON object with
reserved sections for each kind of game text, plus a free `other` section:

```json
{
  "items": {
    "mymod_flask": "Old Flask"
  },
  "other": {
    "mymod_greeting": "Hello!"
  }
}
```

The recognized top-level sections are `items`, `buildings`, `moodles`, `other`,
`character`, and `pauseQuotes`. Any keys you put at the **root** (outside those
sections) are treated as `other` entries. Nested objects are flattened with dots, so
`{"ui": {"title": "..."}}` becomes the key `ui.title`.

## When translations apply

ScavLib injects everything you've registered into the game's locale right after the
game loads a language, so registering in your mod's `OnEnabled` is correct — anything
registered before the first language load is applied when it loads, and anything
registered afterward is flushed immediately.

!!! note "Item names update live on a language switch"

    When the player changes language in-game, ScavLib re-syncs every registered custom
    item's display name to the new language on the spot. So a custom item shows the
    right name immediately after a language switch, without a reload.

## Generating a translation template

Rather than hand-writing the `items` section, you can export a template of every
registered item id and fill in the text. `ExportTemplate` writes a JSON skeleton with
`TODO` placeholders — and if the file already exists, it **merges**, keeping
translations already done and only adding missing keys:

```csharp
using System.IO;
using ScavLib.i18n;

TranslationTemplateGenerator.ExportTemplate(Path.Combine(modFolder, "lang", "EN.json"));
```

Run it once after registering your items to scaffold the file, then hand the JSON to
translators; re-running it later picks up new items without overwriting their work.

## Refreshing UI at runtime

If you change strings while the game is running and want on-screen text to update
immediately, `LocaleManager.RefreshAllUI()` re-applies the current translations to
every localized UI element in the scene. You won't usually need this — the live
re-sync on language change handles the common case — but it's there for dynamic text.

*[LocaleManager]: ScavLib's localization hub — registers strings and injects them into the game's locale.
*[TranslationTemplateGenerator]: Helper that exports a JSON translation template of every registered item id.
*[Locale]: The game's native localization object that ScavLib injects translations into.
*[RegisterItem]: LocaleManager call that registers an item's localized names and descriptions.
*[RegisterString]: LocaleManager call that registers an arbitrary localized string under a key.
*[AutoRegister]: LocaleManager call that loads a mod's lang/*.json files for the current language.
